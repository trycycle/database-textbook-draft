\chapter{関係データモデル}\label{chap:relational-data-model}

第2章では，関係データベースへの入り口として，データベースに求められる役割と関係データベースの特徴について説明しました．
関係データベースは一見すると表の集まりにしか見えませんが，データの正しさを適切に管理するために，背後には「関係データモデル」という理論的基盤を有しています．
本章では，データをモデリングする意義を説明した後，代表的なデータモデルの一つである関係データモデルについて，数学的な観点から説明します．

\section{データのモデリング}
% ---------------------------------------
\subsubsection{モデリングとは？}
科学やビジネスの世界では，モデルあるいはモデリングという用語がしばしば登場します．
ビジネスプロセスモデル，物理モデル，統計モデルなど，世の中には様々なモデルが存在します（図\ref{fig:model}）．

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{figure/relational-data-model/model.jpg}
    \caption{世の中に存在する様々なモデル}
    \label{fig:model}
\end{figure}

ところで，なんとなく言葉にしている「モデル」とは一体何なのでしょうか．
モデルとは，複雑な仕組みや現象，状態などを表現・分析・操作しやすくするために，本質的でない要素を取り除き，関心のある側面のみを抽出し抽象化したものです．
\strong{モデリング}とはモデル化，つまりモデルを作る行為を指します．


\subsubsection{データモデリング}
関係データベースに限らず，データベースで何らかのデータを扱う場合，まずデータをモデリングします．
世の中に存在するデータは多種多様ですから，データを統一的に整理し，計算機で処理しやすい形に抽象化，すなわちモデリングする必要があります．

\strong{データモデリング}とは，データベース化すべき情報を取捨選択し，対象とするデータとその操作に関する枠組み（\strong{データモデル; data model}）を設計する行為です．
対象とする事象やアプリケーションに応じて適切なデータモデルを設計することで，実データを適切に管理・操作することが可能となります．

一般に，データモデルは以下の3つの要素を含みます：
\begin{itemize}
\item データの構造
\item データの制約条件
\item データの操作
\end{itemize}

既に触れたように，本書の主要テーマである\strong{関係データベース（relational database）}は，\strong{関係データモデル（relational data model）}\footnote{関係モデル（relational model）と呼ぶこともある．}に基づき設計されたデータベースです．
関係データモデルは，1970年にEdgar F. Codd氏により提案された，最も代表的なデータモデルです\cite{Codd1970}．
前章で触れたように，関係データモデルではあらゆるデータを\strong{表（table）}としてモデル化します．
一見すると単純な表にしか見えない関係データモデルですが，数学によって定式化されており，強固な理論的基盤を有しています．
関係データモデルを用いることで，データから冗長性を排除し，データを正しく管理することが可能となります．

データベース管理システムで扱われるデータモデルとしては，関係データモデル以外にも以下のようなものがあります：
\begin{itemize}
\item ネットワークモデル\cite{Bachman1969}
\item 階層型データモデル\cite{Tsichritzis1976}
\item オブジェクト指向モデル\cite{Atkinson1994}
\item キー・バリュー（key-value）モデル\cite{DeCandia2007}
\item グラフデータモデル\cite{Angles2008}
\end{itemize}
以降，本章では，数学的な観点から関係データモデルについて説明します．
少々堅い説明にはなりますが，関係データベースという技術が数学に支えられていることを知るためにも，眠気を我慢して読んでください．
関係データモデル以外のデータモデルについては本書では取り扱わないため，興味がある方は\cite{北川本}などを参照してください．

\begin{notebox}{計算機科学とは抽象化の学問}
計算機科学（あるいは情報科学）というと「プログラミングでしょ」と考える方は多いと思いますが，それは間違いです．
計算機科学の本質は，\strong{物事を計算可能な状態にする（抽象化する）ことによって，計算機の上で処理・分析}することです．
データベース技術もその一つです．
計算機科学は抽象化の学問なのです．
\end{notebox}

表\ref{tab:correct-table}は，ある小売店で取り扱っている商品のデータを関係データモデルによって表現したものです．
この表では，各行は小売店で取り扱っている商品に対応し，各列には各商品に関するデータが記述されています．

\begin{table}[h]

\label{tab:correct-table}
\begin{tabular}{@{}llll@{}}
商品 &             &             &              \\ \toprule
\textbf{商品ID} & \textbf{名称} & \textbf{単価} & \textbf{登録日} \\ \midrule

表の左上にある「商品」はこの表の名前を示しており，表の名前は関係データモデル上では\strong{関係名}と呼ばれます．
また，表の行に対応するものを\strong{タプル（tuple）} と呼びます．
タプルは複数のデータ要素の組み合わせを意味し，数学的には$(データ1, データ2, ..., データn)$のような順序付きの組として表現されます．
例えば，関係「商品」の1行のデータは
\begin{equation}
(``P1", ``はーいお茶", 130, 2020/07/15)
\end{equation}というタプルで表現されます．
表の冒頭行には，関係「商品」がどのようなデータを持つかを示す見出しが付けられています．
表における各見出しに対応するものを，関係データモデルでは\strong{属性（attribute）}と呼びます．
また，各タプルにおけるある属性の値を\strong{属性値}と呼びます．
例えば，表\ref{tab:correct-table}の2行目に対応するタプルは
\begin{equation}
(``P2", ``午前の紅茶", 130, 2020/09/25)
\end{equation}
となりますが，このタプルの属性「単価」の属性値は\graybox{130}となります\footnote{表の形式で見ると単なる「行」に見えるタプルですが，データモデル上は「属性値の組み合わせ」として扱われるため，専門的には視覚的な「行」と数学的な「タプル」という語を区別して使います．}．

\subsection{数学における「関係」}
% ---------------------------------------
ところで，関係データベースあるいは関係データモデルの「関係」とは一体何でしょうか．
\strong{関係（relation）} とは，実は数学上の概念です．

集合$S_1, S_2, ..., S_n$が与えられたとき，それらの直積集合$S_1 \times ... \times S_n$とは
\begin{equation}
S_1 \times ... \times S_n = \{(x_1, ..., x_n) \ | \ x_1 \in S_1, ..., x_n \in S_n \}
\end{equation}
と表現されるものです．
（数学上の概念である）「関係」とは，このような直積集合の部分集合を意味します．
正確に書くと，集合$S_1, S_2, ..., S_n$が与えられたとき，直積集合$S_1 \times ... \times S_n$の部分集合を$S_1, S_2, ..., S_n$上の\strong{n項関係}と呼びます．

例えば，$A=\{2, 3\}$，$B=\{-2, -3\}$が与えられたとき，直積$A \times B$は集合AおよびBの要素のすべての組み合わせですから
\begin{equation}
A \times B = \{(2, -2), (2, -3), (3, -2), (3, -3)\}
\end{equation}
となります．
このとき，その部分集合の1つである
\begin{equation}
\{(2, -3), (3, -2)\}
\end{equation}
は集合AおよびB上の（2項）関係となります．
このような（n項）関係の概念を使って，関係データモデルは理論構築されています．


% ---------------------------------------
\subsection{関係データモデルにおける「関係」}\label{subsec:relation}
% ---------------------------------------
関係データモデルにおける\strong{関係（relation）}は「数学における関係」を拡張したものになっており，以下の2つから構成されます．
\begin{itemize}
\item 関係スキーマ
\item インスタンス
\end{itemize}
直感的には，関係スキーマは関係データモデルの「設計図」や「枠組み」であり，インスタンスはその設計図に沿って実際に作られた「具体的なデータの集合」です．
「設計図」や「枠組み」は滅多に変更されませんが，それに沿った「具体的なデータ」は日々追加，更新，削除されていきます．
中身（インスタンス）を正しく管理するためにも，適切な「枠組み」（関係スキーマ）を決める必要があります．
関係データモデルを適切に設計，運用するためにも，関係スキーマとインスタンスの厳密な定義について確認しておきましょう．

\strong{関係スキーマ（relation schema）} とは，関係の名前と属性の集合，および（後述する）一貫性制約の情報を示すものです．
関係スキーマは
\begin{equation}
(\boldsymbol{R}(A_1, ..., A_n), \{\sigma_1, \sigma_2, ..., \sigma_m\})
\end{equation}
の形式で記述されます．
ここで
\begin{itemize}
\item $\boldsymbol{R}$は関係名
\item $A_1, ..., A_n$は属性
\item $\sigma_1, ..., \sigma_m$は一貫性制約
\end{itemize}
に対応します．
一貫性制約が自明な場合やそれを考慮しないときは，関係スキーマを
\begin{equation}
\boldsymbol{R}(A_1, ..., A_n)
\end{equation}
のように簡潔に表記することもあります．
例えば，表\ref{tab:correct-table}の関係「商品」の例の場合，属性（表の見出し）は「商品ID」「名称」「単価」「登録日」となっているので，その関係スキーマは
\begin{equation}
\textbf{商品}（商品ID, 名称, 単価, 登録日）
\end{equation}
となります．

関係スキーマに記された各属性には，属性が取り得る値の集合を定義します．
この集合は\strong{ドメイン（domain; 定義域）} と呼ばれ，属性$A$のドメインは$Dom(A)$と記します．
例えば，表\ref{tab:correct-table}の関係「商品」の例では属性「単価」は金額を表すので，そのドメイン$Dom(単価)$は0以上の整数値が想定されます．
このことを数学的に記すと，以下のようになります\footnote{$\mathbb{N}$は自然数の集合}：
\begin{equation}
Dom(単価) = \{ x \ | \ x \in \mathbb{N} \}
\end{equation}
同様に，属性「商品ID」「名称」「登録日」についても，以下のようにドメインを定義できます：
\begin{align}
&Dom(商品ID) = \{ x \ | x \in Pから始まる文字列集合 \} \\
&Dom(名称) = \{ x \ | \ x \in 文字列集合 \} \\
&Dom(登録日) = \{ x \ | \ x \in 年月日集合（ただし西暦表記） \}
\end{align}

関係を構成するもう一つの要素である\strong{インスタンス（instance）} は，関係スキーマ$\boldsymbol{R}(A_1, ..., A_n)$におけるドメイン$Dom(A_1), ..., Dom(A_n)$の直積の部分集合です．
仮にインスタンスを$I$とすると，数学的には
\begin{equation}
I \subseteq Dom(A_1) \times ... \times Dom(A_n)
\end{equation}
となります．
この数式からも分かるように，インスタンスは前節で説明した数学上の関係に相当します．
また，インスタンス$I$の各要素がタプルとなります．
このように，関係（のインスタンス）は集合として定義されるため，関係同士を結合したり，関係同士の共通部分，和，差などを求めたりする操作を数学的に定義することが可能となります．
これら操作（集合演算や関係演算）については，第\ref{chap:sql-part2}章で説明します．

抽象的な定義の説明が続きましたので，定義と表\ref{tab:correct-table}の例を照らし合わせてみましょう．
表\ref{tab:correct-table}の関係「商品」の関係スキーマは
\begin{equation}
\textbf{商品}(商品ID, 名称, 単価, 登録日)
\end{equation}となることを述べました．
その際，スキーマを構成する各属性のドメインの定義も与えました．
その定義から，例えば属性「単価」の値については
\begin{equation}
Dom(単価) = \{1, 2, 3, .... 10000, .... \}
\end{equation}
のように，あらゆる自然数を取り得ます（そのような値しか取り得ない）．
また，属性「名称」の値については，定義上，
\begin{equation}
Dom(名称) = \{``あ", ``い", ..., ``はーいお茶", ... \}
\end{equation}
のようにあらゆる文字列を取り得ます（そのような値しか取り得ない）．

ドメインの定義を踏まえると，関係「商品」の各属性の直積
\begin{equation}
Dom(商品ID) \times Dom(名称) \times Dom(単価) \times Dom(登録日)
\end{equation}には様々なタプルが要素として含まれることになります．
その中には
\begin{equation}
(``P001", ``こんな商品はありません", 100兆, 2020/01/01)
\end{equation}
のように，実際に商品情報としてありえないタプルも含まれてしまいます．
なぜなら，直積とは集合（ドメイン）の要素のすべての組み合わせだからです．
関係データモデルの「関係」の定義で，関係の構成要素としてドメインの直積ではなく，ドメインの直積の部分集合である「インスタンス」としているのは，取り得るタプルの候補集合（すなわち直積）から，実際にデータとして存在するタプルの集合を選択することに相当します．


\begin{notebox}{タプルは重複が許されない}
一般的に使われる表では，同じ内容を表す行が複数あっても許されます．
一方，関係データモデルに基づく関係（表）では行，すなわちタプルの重複は許されません．
なぜなら，関係データモデルのインスタンスは（数学的な）集合であり，集合は重複を許さないからです．
\end{notebox}

\section{関係データモデルの出発点：第1正規形}
% ---------------------------------------
関係データモデルを使ってデータを正しく管理するためには，どの関係も「第1正規形」という性質を満たしていることが求められます．
第1正規形について説明する前に，原子値の概念について説明します．

関係データモデルにおけるドメインは「属性が取り得る値の集合」ですが，ドメインの要素はそれ以上分解不可能な値を意味する\strong{原子値（atomic value）} であることを想定しています．
原子値の例としては，文字列，整数，実数，真偽値，日付などがあります（図\ref{fig:atomic-value}）．
どれもデータの基本単位になります．
逆に原子値でない例としては，タプルや集合，関係など，複数の独立したデータ要素を組み合わせたデータが挙げられます\footnote{原子値でない値は，非原子値（non-atomic value）またはコンポジット値（composite value; 複合値）と呼ばれることがあります．}．

\begin{figure}[tb]
    \centering
    \includegraphics[width=0.8\textwidth]{figure/relational-data-model/atomic-value.jpg}
    \caption{原子値とそうでない値の例}
    \label{fig:atomic-value}
\end{figure}

ドメインとして原子値しかとらない関係を\strong{第1正規形（first normal form; 1NF）である} と言います．
一方，ドメインとして原子値以外の値をとることを許す関係を\strong{非正規形（unnormalized form）である}と言います\cite{Codd1972}．
Excelなどで表データを扱う場合，表（つまり関係）が第1正規形なのか非正規形なのかを意識することはあまりありませんが，大規模なデータを正しく管理し，適切に集計するには，その違いと影響を理解しておくことが重要です．

\begin{figure}[tb]
    \centering
    \includegraphics[width=1.0\textwidth]{figure/relational-data-model/unnormalized-relation.jpg}
    \caption{非正規関係}
    \label{fig:unnormalized-relation}
\end{figure}
例として，図\ref{fig:unnormalized-relation}の関係を見てみましょう．
図の(a)(b)はともに商品の購買履歴を表していますが，(a)は属性「購入品目」に集合要素が入ってしまっています（例えば，\graybox{\{``はーいお茶'', ``きのこの里'', ``のど飴''\}}）．
(a)の属性「購入品目」のドメインは原子値以外の値（つまり文字列の集合）を許してしまっていることから，(a)は非正規形の関係と見なせます．
一方，図\ref{fig:unnormalized-relation}(b)は(a)の「購買品目」の属性値が原子値（文字列の集合ではなく文字列）になるように設計されているため，第1正規形の関係と見なせます．

データを正しく管理し，適切に集計する上で，なぜ非正規形の関係が都合が悪いのでしょうか．
先の例で考えてみましょう．
図\ref{fig:unnormalized-relation}(a)は非正規形の関係で，1回の購買で複数の商品を購入していた場合，属性「購入品目」のマス目に複数の商品名を入れていました．
一方，(b)は第1正規形となっており，1回の購買で複数の商品を購入していた場合，購入した商品の数だけ行を増やして情報を格納しています．
(a)の方が複数の商品を購入したという情報を1タプル（1行）で表現できていますから，タプル数を抑えられる点では良いように思えますが，データの管理・処理上，実は都合が悪いのです．

例えば，「購買ID」が\graybox{T3}の買い物をしたお客が\graybox{麦茶}を返品したとします．
この時，購買履歴から\graybox{T3}の購買情報を検索し，\graybox{麦茶}に関する情報を削除する必要があります．
第1正規形となっている(b)であれば，上から6行目のタプルを削除するだけで済みます．
一方，非正規形となっている(a)では，「購買ID」の属性値が\graybox{T3}のタプルを検索し，属性「購入品目」の値を\graybox{麦茶, おにぎり}から\graybox{おにぎり}に修正する必要があります．
また，「売上」の属性値も\graybox{250}から麦茶の料金分を差し引きした値に変更する必要があります．
ところが，(a)のみからは麦茶単体の料金が分かりません．
従業員が別途料金表を参照して麦茶の料金を調べ，売上金額を修正することも可能ですが，手間がかかりますし修正ミスも起こり得ます．
一方，第1正規形である(b)の方がタプル数が増えていますが，データの構造が簡潔です．
そのため，データの検索や更新，集計を行うにも，(a)よりも(b)の方がコンピュータで処理しやすいのです\footnote{第\ref{chap:normalization}章で説明しますが，第1正規形には冗長性が残っています．データを正しく管理するためにも，より高レベルの正規形の関係に変換する必要があります．}．

\section{一貫性制約}
% ---------------------------------------
\strong{一貫性制約（integrity constraint）} とは，データベースのデータが対象とする現実世界を正しく反映するように定められた，データが満たすべき規則です．
関係データモデルにおいては，関係スキーマにデータが満たすべき一貫性制約を定義します．
代表的な一貫性制約として以下のようなものがあります\cite{Codd1972}：
\begin{itemize}
\item ドメイン制約
\item キー制約
\item 参照整合性制約
\item データ従属性
\end{itemize}

関係データモデルに基づくデータベース（すなわち，関係データベース）を設計する際には，関係スキーマに一貫性制約を定義し，それに基づき\strong{複数の関係スキーマへの分解}を行います．
一貫性制約を定義すれば，その制約に違反するデータが登録されようとした場合，関係データベース管理システムはエラーを返すなどして，データの登録を拒否します．
こうすることで，データの正しさを担保することができます．

以下，代表的な一貫性制約について説明します．


% ---------------------------------------
\subsection{ドメイン制約}
% ---------------------------------------
\strong{ドメイン制約（domain constraint）} とは，関係スキーマ$\boldsymbol{R}(A_1, ..., A_n)$に従う関係に含まれるタプルの各成分は，対応する属性のドメインの要素でなければならないという制約です．
この制約については，実は属性に関する説明（\ref{subsec:relation}節）ですでに触れています．

ドメイン制約では，各属性のドメインのデータ型（例: 整数，実数，文字列，日付）に加えて，値が取り得る範囲を指定することもあります．
例えば，表\ref{tab:correct-table}でとりあげた関係
\begin{equation}
\textbf{商品}(商品ID, 名称, 単価, 登録日)
\end{equation}
の説明では，属性「単価」や「登録日」のドメインは
\begin{align}
&Dom(単価) = \{ x \ | \ x \in \mathbb{N} \} \\
&Dom(登録日) = \{ x \ | \ x \in 年月日集合（ただし西暦表記） \}
\end{align}
のように定義しました．
これらはまさにドメイン制約です．
これら制約によると，
\begin{itemize}
\item 属性「単価」の値は\strong{必ず}自然数
\item 属性「登録日」の値は\strong{必ず}西暦表記の年月日集合
\end{itemize}
でなければなりません．
よって，表\ref{tab:incorrect-table}に記した表「商品(*)」の各タプルは関係スキーマ$\textbf{商品}(商品ID, 名称, 単価, 登録日)$に従うインスタンスには\strong{なれません}．
なぜなら，
\begin{itemize}
\item 「商品ID」が\graybox{P1}の商品の登録日は年月はあるが日が抜けている
\item 「商品ID」が\graybox{P3}の商品の登録日は和暦で表現されている
\item 「商品ID」が\graybox{P1000}の商品の単価は漢数字（文字列）で表現されている
\end{itemize}
など，関係スキーマ「商品」のドメイン制約に違反しているからです．

\begin{tabular}{@{}llll@{}}

% ---------------------------------------
\subsection{キー制約}
% ---------------------------------------
キー制約の説明をする前に，その前提となる「キー」の概念について説明しておきます．

関係スキーマ$\boldsymbol{R}$における\strong{超キー（super key）} とは，$\boldsymbol{R}$における属性の集合のうち，それらの属性値が決まればおのずと関係スキーマ$\boldsymbol{R}$のインスタンス中のタプルが唯一ひとつに決まる（タプルを一意に特定できる）ものを指します．
これだけ聞いても何のことかよく分からないので，例を見てみましょう．

表\ref{tab:contact-relation}は関係スキーマ
\begin{equation}
\textbf{連絡先}(学生ID, 名前, 学年, 大学email, 自宅住所)
\end{equation}
に従うデータを表形式で記したものです．
各行（タプル）は学生の連絡先情報を示しています．
大学において学生情報を管理する場合，学生ID（の値）が決まれば\strong{唯一1名の学生の連絡先情報}（行; タプル）を照会できるよう，学生IDは重複がないように設定します．
このとき，この関係スキーマ「連絡先」において属性「学生ID」は超キーとなります．
同様に，属性「大学email」も超キーとなります．
なぜなら，通常，同じ大学emailを複数の学生に割り当てることはないからです．

\begin{tabular}{@{}lllll@{}}

なお，\{ 学生ID, 名前 \}，\{ 学生ID, 名前, 学年 \} のように学生IDを含む属性の集合も超キーになります．
例えば，関係「連絡先」において属性「学生ID」の値が\graybox{S1}，属性「名前」の値が\graybox{川澄桜}であるタプルは唯一1つに決まるので，属性集合 \{ 学生ID, 名前 \} も超キーになります．
これは属性「学生ID」が超キーですから，自明です（学生IDの値が決まった時点でタプルが特定できる）．

一方，属性「自宅住所」は超キーにはなり得ません．
属性「学生ID」の値が\graybox{S3}と\graybox{S4}の学生の住所が同じ（つまり同じところに住んでいる）で，「自宅住所」の値を指定しても連絡先タプルが一意に特定できないからです．

上の例でも明らかなように，その定義上，超キーは複数ありえます．
しかし，色々候補が考えられるからといって，タプルを一意に特定する手がかりとして，属性の複雑な組み合わせを使うのは避けたいです．
例えば，上の関係「連絡先」の例では属性「学生ID」も属性集合 \{学生ID, 名前\} も超キーとなりますが，学生IDだけで連絡先を特定できるのであれば，わざわざ名前「も」使ってタプルを特定しようとするのは無駄です．

そこで，超キーのうち，そこからどの属性を取り除いても超キーとしての性質を失うものを\strong{候補キー（candidate key）} と定義します．
候補キーは単純に\strong{キー（key）} と呼ばれることもあります．
例えば，先の例の関係スキーマ「連絡先」では，属性「学生ID」や「大学email」が候補キーとなりえます．

ここで，キー制約を説明するための最後のピースとして，主キーの概念を導入します．

関係データモデルでは属性の値はドメイン制約によって制限を受けますが，そもそも属性値が「存在しない」あるいは「未定義である」という状態もあり得ます．
このように，「値が存在しない」「値が不明である」という特別な状態を示す値は\strong{NULL値あるいは空値} と呼ばれます\footnote{0（ゼロ）や``''（空文字）はそれ自体に情報があるのでNULL値ではありません．}．
ドメイン制約は属性値が存在する場合の制約であるため，関係内のタプルには\graybox{NULL}が含まれる可能性があります．
すなわち，候補キーの属性値が\graybox{NULL}であるタプルが存在する可能性があります．

そこで，候補キーのうち
\begin{itemize}
    \item 値としてNULL値を取る可能性がなく，かつ
    \item データベース管理上都合のよい候補キー
\end{itemize}
の1つを\strong{主キー（primary key）}と定義します\footnote{主キー以外の候補キーは\strong{代替キー}と呼ばれます．}．
例えば，先の関係スキーマ「連絡先」の場合，候補キーは「学生ID」と「大学email」でしたが，大学emailは「氏名\@xxx.ac.jp」のようなものから「学籍番号\@xxx.ac.jp」のようなものに変更される可能性がありますし，学生が大学を卒業した後は使えなくなる（削除され，情報がNULL値になる）のが一般的です．
一方，学生IDは学籍番号のようなものですから，入学時にどの学生にも必ず割り振られますし，一度決まったら変わりません．
また，卒業後も成績証明書の発行などで利用する可能性もありますから，永続的なものと考えられます．
これらを踏まえると，関係スキーマ「連絡先」においては属性「学生ID」を主キーとして選ぶのがよさそうです．

\textbf{科目ID} & \textbf{学生ID} & \textbf{成績} \\ \midrule

なお，候補キーの定義から，主キーを単一の属性ではなく，属性の集合とすることもできます．
例えば，表\ref{tab:course-completion-relation}は，学生が履修した科目の成績をあらわす関係「履修」を表として示したものです．
この表において．科目IDあるいは学生IDだけでは履修のタプルが特定できませんが，科目IDと学生IDの両方が決まれば，どの履修タプルを指しているのか（どの学生が履修したどの科目の情報か）を特定することができます．
それゆえ，関係スキーマ「履修」においては \{ 科目ID, 学生ID \} の属性ペアが主キーとなります．
このような属性の集合からなる主キーは\strong{複合主キー（composite primary key）} と呼ばれます．

ここで一度キーの概念を整理しておきましょう．
主キー，候補キー，超キーの関係は，図\ref{fig:key-concept-mapping}のようにまとめることができます．
\begin{figure}[h]
    \centering
    \includegraphics[width=1.0\textwidth]{figure/relational-data-model/key-concept-mapping.jpg}
    \caption{キーの概念の整理}
    \label{fig:key-concept-mapping}
\end{figure}

超キー，候補キー，主キーの概念を押さえたので，本節の主題であるキー制約について学びましょう．
\strong{キー制約（key constraint）}とは，関係$\boldsymbol{R}$のスキーマに主キーや候補キーが設定されたとき，関係$R$のインスタンスにおいては
\begin{itemize}
\item 主キーあるいは候補キーに設定された属性（あるいは属性の集合）の値は重複があってはならない
\item その要素（タプル）は主キーあるいは候補キーによって一意に特定されなければならない
\item 主キーとなる属性の値は\strong{NULL値} であってはならない
\end{itemize}
という条件を満たすべしという制約です．
小難しく見える定義ですが，要するにキー制約は「ある属性が主キー（あるいは候補キー）と設定されたら，それがきちんと主キー（あるいは候補キー）の役割を果たすようなデータになっていなければならない」ということを意味しています．
関係に主キーや候補キーを設定するということは，その関係にキー制約を与えるということと同義です．

ところで，関係スキーマ$\boldsymbol{R}$において設定したい一貫性制約が複数あって，それらが$\sigma_1, \sigma_2, ..., \sigma_m$であるとき，関係スキーマは$(\boldsymbol{R}(A_1, ..., A_n), \{\sigma_1, \sigma_2, ..., \sigma_m\})$のように記すと述べました．
この際，キー制約以外の一貫性制約には注目しない場合は，以下のように関係スキーマの主キーに下線を引いてキー制約を表します．
\begin{equation}
\boldsymbol{R}(\underline{A_1, A_2}, ..., A_n)
\end{equation}
例えば，図\ref{fig:key-concept-mapping}の関係「連絡先」であれば，キー制約だけに着目すると，その関係スキーマは以下のように表せます．
\begin{equation}
\textbf{連絡先}(\underline{学生ID}, 名前, 学年, 大学email, 自宅住所)
\end{equation}

\begin{notebox}{キーは関係スキーマに対して設定される}
キーは，関係データベース内にあるタプル（行）だけを見て決めてはいけません．
例えば表\ref{tab:contact-relation}に示した関係「連絡先」の場合，表の見えているところだけに注目すると，名前を特定すれば連絡先タプルを特定できるので，属性「名前」はキーになりえるように思えます．
しかし，現実的には同姓同名の学生も存在しうるため，将来的に関係「連絡先」に同姓同名の学生が登録されるかもしれません．
つまり，属性「名前」をキーにすると，連絡先のタプルを一意に特定できない可能性があります（キーの性質を失う）．

関係データモデルの設計では，どのようなデータをデータベースに格納する「可能性があるか」を事前に想定して関係スキーマを設定する必要があります（その中にキーの設定も含まれる）．
インスタンスを見て，キーを決めてはいけません．
データは関係スキーマに従って格納されるべきだからです．
キーは「\strong{関係スキーマに対して設定される}」ことを意識しましょう．
\end{notebox}


% ---------------------------------------
\subsection{参照整合性制約}
% ---------------------------------------
関係データモデル（およびそれに基づく関係データベース）では，対象となる事象のデータを正しく管理するために，\strong{データを複数の関係（表）に分けて管理する}ことがほとんどです．
例えば，図\ref{fig:score-db}はある大学における学生の成績を管理するための関係データベースですが，学生情報や科目情報，成績情報を管理するために，データが複数の関係に分かれています．

\begin{figure}[tb]
    \centering
    \includegraphics[width=1.0\textwidth]{figure/relational-data-model/score-db.jpg}
    \caption{成績管理データベース}
    \label{fig:score-db}
\end{figure}

ところで，図\ref{fig:score-db}の関係「履修」の属性「科目ID」「学生ID」は，その値だけ見ても具体的に何を意味しているのかが分かりません．
これら値の意味を読み解くためには
\begin{itemize}
\item 「科目ID」が何の科目のことを指しているのか
\item 「学生ID」がどの学生のことを指しているのか
\end{itemize}
を把握するための情報を管理している他の関係（表）が必要となります．
幸い，図\ref{fig:score-db}の関係データベースには，関係「履修」以外にも関係「科目」および関係「学生」があります．
それゆえ，これら3つの関係（表）がどのように関連付けられているかを理解すれば，関係「履修」の属性「科目ID」「学生ID」の値が具体的に何を意味しているのかを把握することができます．
例えば，関係「履修」には属性「科目ID」の値が\graybox{C1}で，属性「学生ID」の値が\graybox{S1}のタプルがありますが，このタプルが具体的に何を意味しているのか，把握してみましょう．
それには
\begin{itemize}
\item 関係「学生」から「学生ID」が\graybox{S1}のものを探し
\item 関係「科目」から「科目ID」が\graybox{C1}のものを探す
\end{itemize}
ことで，注目しているタプルは「\graybox{A学部}の\graybox{川澄桜}さんが履修した\graybox{2023}年度開講の\graybox{線形代数}の成績は\graybox{不可}だった」ということが把握できます．

ところで，関係「履修」のタプルにある属性「科目ID」の\graybox{C1}という値が，関係「科目」のインスタンスの属性「科目ID」に含まれていなければならない（関係「科目」の列「科目ID」にその値が存在しなければならない），ということは自明でしょう．
関係「履修」の中にはあっても，関係「科目」には存在していなければ，成績管理として破綻していることになるからです．
これは「学生ID」の属性値\graybox{S1}についても同様です．
この例では
\begin{itemize}
\item 関係「履修」の属性「学生ID」と関係「学生」の主キーである属性「学生ID」
\item 関係「履修」の属性「科目ID」と関係「科目」の主キーである属性「科目ID」
\end{itemize}
が紐付いています．
言い方を変えると，関係「履修」は属性「学生ID」および「科目ID」の値を通して，関係「学生」および「科目」の情報を参照していることになります．

このように，関係スキーマ$\boldsymbol{R_1}(..., FK, ...)$と$\boldsymbol{R_2}(\underline{PK}, ...)$が与えられ，$\boldsymbol{R_1}$におけるタプルの属性$FK$の値が$\boldsymbol{R_2}$におけるいずれかのタプルの主キー$PK$の値と一致するように設計されているとき，$\boldsymbol{R_1}$の$FK$を$\boldsymbol{R_2}$の$PK$に対する\strong{外部キー（foreign key）} と呼びます．
関係スキーマ上では，関係スキーマ$\boldsymbol{R_1}$の属性$FK$が関係スキーマ$\boldsymbol{R_2}$の主キー$PK$に対する外部キーであることを，以下のように記します：
\begin{equation}
%\boldsymbol{R_1}.FK \subseteq \boldsymbol{R_2}.PK \\
\pi_{FK}(\boldsymbol{R_1}) \subseteq \pi_{PK}(\boldsymbol{R_2})
\end{equation}
なお，$\pi_{A}(\boldsymbol{R})$は，関係スキーマ$\boldsymbol{R}$に従う関係のインスタンスから属性$A$の値だけを取り出した（射影した）集合を表します．

図\ref{fig:score-db}の例では，関係スキーマ「履修」の属性「学生ID」と「科目ID」が外部キーとなっています．
これを数学的には
\begin{align}
%履修.学生ID \subseteq 学生.学生ID \\
\pi_{学生ID}(\textbf{履修}) \subseteq \pi_{学生ID}(\textbf{学生}) \\
%履修.科目ID \subseteq 科目.科目ID
\pi_{科目ID}(\textbf{履修}) \subseteq \pi_{科目ID}(\textbf{科目})
\end{align}
と記します．

前置きが長くなりましたが，本節のテーマである\strong{参照整合性制約（referential constraint）}\footnote{「参照制約」と呼ぶこともあります．} とは「関係スキーマにおいて外部キーが設定されたとき，そのいかなるインスタンスも設定された外部キーの条件を満たさなければいけない」という制約です．
関係に外部キーを設定するということは，参照整合性制約を与えているということになります．

% ---------------------------------------
\subsection{データ従属性}
% ---------------------------------------
これまで一貫性制約として，ドメイン制約，キー制約，参照整合性制約を挙げてきました．
定義の話が続きましたのでこの3つの制約でもお腹いっぱいかも知れませんが，関係データモデルにはこれらの制約に加え，データ間に成立する制約を定義するための重要な概念があります．
それが\strong{データ従属性（data dependency）}です．
データ従属性としては，関数従属性\cite{Armstrong1974}，多値従属性\cite{Fagin1977}，結合従属性など，様々なものが提案されています．

\strong{関数従属性（functional dependency）}は，データ従属性の中でも最も単純で，かつ実用的なものです．
関数従属性は，「関係$R(..., X, ..., Y, ...)$において，属性（あるいは属性集合）$X$の値が決まると属性$Y$の値も一意に決まる」という性質です．
言い方を変えると，「$X$の値が同じであれば，対応する$Y$の値も必ず同じになる」ということを意味します．
このことを数学的には
\begin{equation}
X \to Y
\end{equation}
と記します．

例えば，関係スキーマとして
\begin{equation}
\textbf{連絡先}(\underline{学生ID}, 名前, 学年, 大学email, 郵便番号, 都道府県, 自宅住所)
\end{equation}
が与えられたとしましょう．
一般常識から，住所が決まればそれが存在する都道府県や郵便番号もひとつに決まります．
このことから，関係スキーマ「連絡先」においては
\begin{align}
自宅住所 \to 郵便番号 \\
自宅住所 \to 都道府県
\end{align}
という関数従属性が定義できます．
関数従属性は，主キーが決まればタプルが一意に決まるという「キー制約」を一般化したものと捉えることもできます．

関数従属性を含めたデータ従属性は，データの更新があったときにその影響が最小限になるように関係データベースを設計する上で，極めて重要となる概念です．
ですので，特に関数従属性については別章で改めて詳しく説明します．

\subsubsection{Q6. 超キーと候補キー}

    \item $\{A, B\}$
    \item $\{A, C\}$
    \item $\{A, B, C\}$
    \item $\{A, B, C, D\}$
\end{enumerate}


\subsubsection{Q7. 候補キー2（令和4年度 ITパスポート試験 問65改題）}
関係スキーマ
\begin{equation}
\textbf{従業員}(従業員番号, 従業員名, 部門コード, 生年月日, 住所)
\end{equation}
について考えます．
関係「従業員」では，以下のような制約条件が成り立ちます：
\begin{itemize}
\item 各従業員は重複のない従業員番号を1つだけもつ
\item 同姓同名の従業員がいてもよい
\item 各部門は重複のない部門コードを1つだけもつ
\item 1つの部門には複数名の従業員が所属する
\item 1人の従業員が所属する部門は1つだけである
\end{itemize}
\begin{enumerate}
    \item 上記制約から成立する関数従属性を$X \to Y$の形式ですべて列挙しなさい．
    \item 関数従属性に基づき，関係「従業員」の候補キーをすべて列挙しなさい．
\end{enumerate}
