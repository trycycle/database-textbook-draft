\chapter{関係データベースの設計と問い合わせの例 -- フリマアプリのデータ管理と分析}
これまでの章では，関係データモデルの基本概念，実体関連モデルと正規化理論による関係データベースの設計方法，SQLによる関係データベースへの問い合わせ手法，問い合わせ高速化のための索引づけについて説明してきました．
最終章では，断片的に学んできた関係データベースの知識を総動員して，ある関係データベースの設計と問い合わせの例を見ていきましょう．

\section{問題設定}
最終章でも山畑さんに登場してもらいましょう．
以下は，架空の人物である山畑さんのその後のストーリーです．

\begin{framed}
ショッピングサイトの運営を始めて6年．運営開始当初の山畑さんは，データベースといえば「大量のデータの集まり」くらいの認識しかなく，Excelシートに蓄積された販売履歴は「データベース」と呼ぶにはあまりにもずさんなものだった．
データには誤りも多く，例えば販売価格がマイナスになっているレコードもあった．
さらに，同じ商品なのに販売単価が異なるなど，データ間で食い違いがあるレコードも散見された．

そんな「データベース（笑）」を使ってデータ分析を行おうとしていた自分を恥ずかしく思えるほど，今の山畑さんは関係データベース設計についての確かな知識と経験を身につけている．
ショッピングサイトに関係データベースを導入して以降，販売履歴をはじめとする様々なデータを正しく管理できるようになった．
また，蓄積された大規模データと分析技術を駆使して，様々なAIサービスも提供できるようになった．
今では，本業のショッピングサイト事業とは別に，データベース設計の受託業務も行うようになり，データベースを活用したウェブサイト作りの巧さではちょっとした話題になっている．

ある日のこと，そんな絶好調の山畑さんのところに，母校の学生起業家サークルの代表を務める後輩から「大学生を対象としたフリマアプリ作り」に関する相談が舞い込んだ．
彼曰く，物価高の影響から，大学キャンパス内で使わなくなった教科書や家電，服などを学生間で売買する活動が非常に盛んになっているという．
しかし，既存の学内掲示板やSNSでのやり取りでは管理に限界があり，専用のフリマアプリを立ち上げたいとのことだった．

さらに彼は，単なる売買アプリにとどまらず，ゆくゆくは蓄積したデータを使って「どんな商品が売れやすいのか」「いくらで出品するのが適正か」をサジェストするAI機能も組み込みたいと熱く語った．しかし，
いざ開発を始めようとしたものの，アプリの裏側を支えるデータベースの設計，特に多様な商品をどう管理すればよいのかに頭を悩ませており，実績のある山畑さんに助けを求めてきたのだった．

データベースの設計と分析も生業にしつつある山畑さん．
後輩の熱意と，データ分析のやりがいがありそうな要件に惹かれ，データベース設計の依頼を二つ返事で引き受けたのだった ．
\end{framed}

データベース設計の依頼を引き受けた山畑さんは，要件定義を行うために，サークルの開発メンバーたちにヒアリングを行いました．
以下は，ヒアリングの結果です．
\begin{itemize}
\item アプリの利用者は事前にユーザ登録を行う必要がある．登録時には，ユーザID，年齢，性別，メールアドレスを記録する．利用者は，商品の出品者になることもあれば購入者になることもある．
\item ユーザはアプリ上に商品を出品できる．出品時には，商品ID，設定価格，商品の状態（新品，美品，傷ありなど），および説明文が登録される．
\item 商品を探しやすくするため，商品は「書籍・エンタメ」「ファッション」「家電・スマホ」といった大カテゴリと，さらに細かい「専門書」「小説」「メンズアパレル」「生活家電」といった小カテゴリに体系づけて分類したい．
\item 配送方法（ポスト投函，宅急便，着払いなど）や基本送料のルール，どちらが送料を負担するかという情報も商品ごとに設定される．
\item \textbf{【後輩が特に悩んでいた点】} 出品される商品には，カテゴリごとに特有の入力項目がある．例えば，本ならISBNや出版社，著者，服ならサイズやブランド，家電なら保証期間や型番などを登録させたい．しかし，これらすべてを同じ「商品」のテーブルに詰め込むと，本を出品した際にはサイズや保証期間の列が空欄（NULL）だらけになってしまい，データベースとして無駄が多く不格好になるのではないかと懸念している．
\item 出品された商品に対して，他のユーザは「いいね」をつけて保存することができる．
\item 商品が購入されると「取引」が成立する．取引には，取引ID，対象となった商品，購入したユーザ，および取引日時が記録される．
\item 取引完了後，購入者は出品者に対して1〜5の評価スコアとコメントをつける（レビュー）．このレビューの蓄積から計算される「出品者の平均評価スコア」が，ユーザの信頼度を表す重要な指標となる．
\item \textbf{【将来のAI分析に向けた展望】} 将来的には，商品の状態や説明文の長さ，送料負担の有無，そして出品者の平均評価スコアなどの要因が，商品の売れやすさにどう影響するかを機械学習を用いて分析し，ユーザに最適な出品方法をアドバイスできるようにしたい．
\end{itemize}


\section{実体関連図の作成}
ヒアリングの結果をもとに，実体関連図を作成してみましょう．
実体関連図の作成にあたって，まずは\ref{sec:what-is-er-model}節のコラムで触れた「言語分析に基づく実体関連モデルの抽出法」を使って実体型，関連型，それらの属性の候補を洗い出してみます．

ヒアリングの文章から，システムで管理すべきデータに関わる名詞を抽出すると，「ユーザ」「ユーザID」「年齢」「性別」「メールアドレス」「出品者」「購入者」「商品」「商品ID」「価格」「（商品の）状態」「説明文」「大カテゴリ」「小カテゴリ」「配送方法」「基本送料」「送料負担（の有無）」「ISBN」「出版社」「著者」「サイズ」「ブランド」「保証期間」「型番」「いいね」「取引」「成立」「取引ID」「レビュー」「評価スコア」などが挙げられます．

ここで注目すべきは，商品の多様性の扱いです．
ヒアリング結果において，後輩が「カテゴリごとに特有の入力項目をどう管理するか」を懸念していた点に着目しましょう．
アプリでは，すべての商品に共通する属性（価格や状態など）に加え，書籍ならISBN，服ならサイズ，家電なら保証期間といった特有の属性が存在します．
これらを一つの実体型にまとめると，多くの列が空欄（NULL）になってしまいます．

そこで，\ref{sec:inheritance}節で説明した汎化・継承の概念を用いましょう．
共通の属性を持つ\strong{上位実体型として「商品」を定義し，その下位実体型として各カテゴリ特有の属性を持つ「書籍」「アパレル」「家電」を定義}するのが効果的です．

一方，「ユーザ」についてはどうでしょうか．
ヒアリングでは「出品者」や「購入者」という言葉が登場しますが，「一人の利用者が両方の役割を担う」可能性があります．
そのため，役割ごとに実体を分けるのではなく，単一の「ユーザ」実体型として定義するのが良さそうです．

さらに，「大カテゴリ」「小カテゴリ」「配送方法」「基本送料」といった情報ですが，今回はひとまず，実体型「商品」を特徴づける属性として扱ってみることにしましょう．
後ほど説明しますが，実はこの設計には少し問題が潜んでいます．
しかし，まずはこのまま実体関連図を完成させてみます．

「評価スコア」の扱いも重要です．
これは取引後のレビューの積算から計算される動的な属性であるため，実体「商品」や「ユーザ」の静的な属性としては持たせず，独立した「レビュー」実体型に紐づくデータとして管理するのが良さそうです．
これは，データ従属性やデータの一貫性を保つための設計上の判断です．

これらを踏まえ，独立した存在として識別可能な「実体型」とその「属性」を整理すると以下のようになります．
\begin{itemize}
    \item ユーザ：ユーザID（主キー），年齢，性別，メールアドレス
    \item 商品（上位実体）：商品ID（主キー），大カテゴリ，小カテゴリ，価格，状態，説明文，配送方法，基本送料，送料負担
    \begin{itemize}
        \item 書籍（下位実体）：ISBN，出版社，著者
        \item アパレル（下位実体）：サイズ，ブランド
        \item 家電（下位実体）：保証期間，型番
    \end{itemize}
    \item 取引：取引ID（主キー），取引日時
    \item レビュー：レビューID（主キー），評価スコア，コメント
\end{itemize}

次に，これらの実体間のつながりを表す「関連型」を，動詞に注目して整理します．
\begin{itemize}
    \item 出品：ユーザと商品の関連．一人のユーザは複数の商品を出品でき，各商品は一人のユーザによって出品されるため，1対多の関連となります．
    \item いいね：ユーザと商品の関連．ユーザは複数の商品に「いいね」ができ，商品も複数のユーザから「いいね」を受けるため，多対多の関連となります．
    \item 成立：n項関連の概念を用い，実体型「取引」「ユーザ」「商品」の3つを結びます．商品の購入という事象は「誰が（ユーザ）」「何を（商品）」「いつ買ったか（取引）」が同時に決定することで成立します．この3項関連においては，「取引」の実体が一つ決まれば，関連型「成立」を通して対応する「ユーザ」と「商品」がそれぞれ一意に決まるという強い制約を持たせます．
    \item 評価：取引とレビューの関連．一つの取引に対して一つのレビューが行われるため，1対1の関連となります．
\end{itemize}

\section{実体関連図から関係スキーマへの変換}
次に\ref{sec:erd-to-relational-model}節で説明した方法を用いて，実体関連モデルを関係データベースで扱うための関係スキーマ（テーブルの定義）へと変換していきます．

まず，汎化・継承を用いた実体型の変換です．
属性の違いを明確に分離し，データの冗長性を防ぐため，別々の関係スキーマに変換するアプローチを採用します．
下位実体型である「書籍」「アパレル」「家電」の関係スキーマは，上位実体型である「商品」の関係スキーマの主キー（商品ID）を外部キーとして受け継ぎます．
独立管理される実体型「ユーザ」「商品」「取引」「レビュー」は，実体関連図上の属性をそのまま関係スキーマの属性とします．

次に，関連型の変換です．
関連型は多重度によって変換ルールが異なることを思い出して，すべての関連型を関係スキーマに変換してみましょう．
「多対多」の関連である「いいね」については，両者の主キーを組み合わせた関係スキーマを作成します．
「1対多」および「1対1」の関連については，「多」側（あるいは特定の1側）の主キーを関連型の主キーとし，つながる実体型の主キーを外部キーとして持たせます．
具体的には，以下のようになります．
\begin{itemize}
\item 「1対多」の関連である「出品」：「多」側に対応する実体型の主キー（商品ID）を主キーとし，つながる「ユーザ」の主キー（ユーザID）を外部キーとして持たせます．
\item 「1対1」の関連である「評価」：今回は実体型「レビュー」の主キー（レビューID）を「評価」の主キーとし，つながるもう一方の実体型「取引」の主キー（取引ID）を外部キーとして持たせます．
\item 3項関連である「成立」：商品の購入という事象は取引によって一意に決まるため，「取引」の実体型に対応する主キー（取引ID）を主キーとし，つながる「ユーザ」と「商品」の主キー（ユーザID，商品ID）を外部キーとして持たせます．
\end{itemize}

以上，実体関連図を関係スキーマに変換した結果は以下のようになります（簡潔にするため，参照整合性制約を省略しています）．
\begin{align}
&\textbf{ユーザ}(\underline{\text{ユーザID}}, \text{年齢}, \text{性別}, \text{メールアドレス}) \\
&\textbf{商品}(\underline{\text{商品ID}}, \text{価格}, \text{状態}, \text{説明文}, \text{大カテゴリ}, \text{小カテゴリ}, \text{配送方法}, \text{基本送料}, \text{送料負担}) \\
&\textbf{書籍}(\underline{\text{商品ID}}, \text{ISBN}, \text{出版社}, \text{著者}) \\
&\textbf{アパレル}(\underline{\text{商品ID}}, \text{サイズ}, \text{ブランド}) \\
&\textbf{家電}(\underline{\text{商品ID}}, \text{保証期間}, \text{型番}) \\
&\textbf{取引}(\underline{\text{取引ID}}, \text{取引日時}) \\
&\textbf{レビュー}(\underline{\text{レビューID}}, \text{評価スコア}, \text{コメント}) \\
&\textbf{いいね}(\underline{\text{ユーザID}, \text{商品ID}}) \\
&\textbf{出品}(\underline{\text{商品ID}}, \text{ユーザID}) \\
&\textbf{評価}(\underline{\text{レビューID}}, \text{取引ID}) \\
&\textbf{成立}(\underline{\text{取引ID}}, \text{ユーザID}, \text{商品ID})
\end{align}

さて，すべての実体と関連を関係スキーマに変換できましたが，このままではテーブルの数が多く，データを参照するたびに何度もテーブルの結合を行わなければなりません．
ここで，変換された関係スキーマの「主キー」に注目してみましょう．

関係スキーマ「商品」と「出品」は，どれも同じ主キー（商品ID）を持っています．
また，「取引」と「成立」，および「レビュー」と「評価」もそれぞれ同じ主キー（取引ID，レビューID）を持っています．
主キーが同じということは，それらのレコードが1対1で対応していることを意味します．
そのため，同じ主キーを持つ関係スキーマの属性を1つの関係スキーマに統合（包含）してしまったほうが，実務的には管理コストが下げられましし，テーブルの結合頻度も小さくできます\footnote{実用的な観点からテーブルを使う頻度とパフォーマンスを考慮して，テーブル「商品」と「出品」を統合した方がよいと判断しましたが，概念モデルの観点からは2つのテーブルは分けておいたほうが意味的な一貫性を維持できます．}．

これらを統合した結果，関係スキーマは以下のようにすっきりと整理されます（簡潔にするため，参照整合性制約を省略しています）．
なお，分かりやすくするため，統合した属性名には「出品者\_」「購入者\_」などを付与しています．
\begin{align}
&\textbf{ユーザ}(\underline{\text{ユーザID}}, \text{年齢}, \text{性別}, \text{メールアドレス}) \\
&\textbf{商品}(\underline{\text{商品ID}}, \text{価格}, \text{状態}, \text{説明文}, \text{大カテ}, \text{小カテ}, \text{配送方法}, \text{基本送料}, \text{送料負担}, \text{出品者\_ユーザID}) \\
&\textbf{書籍}(\underline{\text{商品ID}}, \text{ISBN}, \text{出版社}, \text{著者}) \\
&\textbf{アパレル}(\underline{\text{商品ID}}, \text{サイズ}, \text{ブランド}) \\
&\textbf{家電}(\underline{\text{商品ID}}, \text{保証期間}, \text{型番}) \\
&\textbf{取引}(\underline{\text{取引ID}}, \text{取引日時}, \text{購入者\_ユーザID}, \text{商品ID}) \\
&\textbf{レビュー}(\underline{\text{レビューID}}, \text{評価スコア}, \text{コメント}, \text{取引ID}) \\
&\textbf{いいね}(\underline{\text{ユーザID}, \text{商品ID}})
\end{align}


\section{正規化理論に基づく関係スキーマの修正}
作成した関係スキーマを用いれば，データベースからデータの冗長性が排除され，更新時異状が発生しないか，\ref{chap:normalization}章で学んだ正規化理論の観点から確認しましょう．
具体的には，関数従属性に注目して，関係スキーマがボイス・コッド正規形もしくは第3正規形を満たしているかチェックします．

ここで，関係スキーマ「商品」に注目してみましょう．
このスキーマには，商品の分類に関する情報（大カテゴリ，小カテゴリ）や配送に関する情報（配送方法，基本送料，送料負担）が含まれています．
さて，「商品ID」は主キーに設定されているため，その値が決まれば，商品の「小カテゴリ」と「配送方法」が一意に決まります．
また，「小カテゴリ」が決まれば「大カテゴリ」が一意に決まり，「配送方法」が決まれば「基本送料」と「送料負担」が一意に決まります．
つまり，関係スキーマ「商品」においては，以下のような関数従属性が成立しています．
\begin{align}
&FD_1: \text{商品ID} \to \text{\{小カテゴリ，配送方法\}} \\
&FD_2: \text{小カテゴリ} \to \text{大カテゴリ} \\
&FD_3: \text{配送方法} \to  \text{\{基本送料，送料負担\}}
\end{align}

関数従属性$FD_2$の左辺である属性「小カテゴリ」，$FD_3$の左辺である属性「配送方法」は，関係スキーマ「商品」の超キーではありません．
このように，主キーに直接従属しているのではなく，別の属性を経由して間接的に従属している関数従属性が残っている関係スキーマは，ボイス・コッド正規形も第3正規形も満たしていません．

もしこのままの設計でシステムを運用し，例えば配送会社の運賃改定によって基本送料が変更されたとします．
このとき，その配送方法が設定されているすべての商品のレコードを探し出し，一つ残らず「基本送料」の値を更新しなければなりません．
もし更新漏れがあると，同じ配送方法なのに商品によって送料が異なるという更新時異状が発生してしまいます．

このような異状を防ぐため，\ref{sec:FD-lossless-decomposition}節で解説した情報無損失分解の定理に従い，関数従属性の原因（小カテゴリ，配送方法）と結果（大カテゴリ，基本送料，送料負担）のセットからなる関係スキーマと，元のスキーマから結果の属性だけを切り離した関係スキーマに分解します．
具体的には，以下のように関係スキーマ「商品」から「小カテゴリ」および「配送方法」をそれぞれ主キーとする新しい関係スキーマを切り出します．
\begin{align}
&\textbf{商品}(\underline{\text{商品ID}}, \text{価格}, \text{状態}, \text{説明文}, \text{小カテゴリ}, \text{配送方法}, \text{出品者\_ユーザID}) \\
&\textbf{カテゴリマスタ}(\underline{\text{小カテゴリ}}, \text{大カテゴリ}) \\
&\textbf{配送方法マスタ}(\underline{\text{配送方法}}, \text{基本送料}, \text{送料負担})
\end{align}
このように分解することで，送料が変更されても「配送方法マスタ」テーブルの該当レコードを1箇所更新するだけで済み，更新時異状が発生しないようになります．

なお，要件の最後にあった「出品者の平均評価スコア」についてですが，平均値などのデータの集計結果は，独立したテーブルとしては持たせず，SQLの集約関数などを用いて算出するのが定石です．

こうして，すべての要件を満たしつつ，データの重複がなくなり，データを正しく矛盾なく管理・処理できる堅牢な関係データベースの設計が完成しました．
最終的に完成した関係スキーマは以下のようになります．
正確さのため，各スキーマ上で考慮すべき参照整合性制約も記しています．
これら関係スキーマを用いることで，山畑さんの後輩はデータを正しく管理することができる関係データベースを立ち上げることができるようになりました．

&\textbf{ユーザ}(\underline{\text{ユーザID}}, \text{年齢}, \text{性別}, \text{メールアドレス}) \\[7pt]

\subsection{予測モデル構築のためのデータ抽出}\label{sec:data-extraction}
最後に，後輩の要望の目玉であった「商品の売れやすさを予測するAI」を作るためのデータセットを抽出します．
機械学習モデルにデータを読み込ませるためには，商品情報，カテゴリ，送料，そして先ほど計算した出品者の評価や「いいね」の数などを1つの表にまとめる必要があります．

ここで注意しなければならないのが，複数のテーブルの結合による「行の増殖」です．
もし，「商品」テーブルに対して「レビュー」のテーブルと「いいね」のテーブルを単純に結合（\graybox{JOIN}）してしまうと，1つの商品に複数の「いいね」と複数の「レビュー」が存在した場合，掛け算でレコード数が増殖し，正しい集計ができなくなってしまいます．

これを防ぐためには，\ref{sec:subquery}節で学んだ副問い合わせ（\graybox{WITH}句）を用いて，「出品者ごとの平均評価」と「商品ごとのいいね数」をそれぞれ別々に事前集計しておくのが定石です．

抽出を行うSQLはコード\ref{code:machine-learning-dataset-sql}のようになります．
なお，まだ評価がない出品者のスコアや「いいね」が1件もついていない商品のいいね数，まだ売れていない商品の取引IDなどは，結合時に値が存在しない（\graybox{NULL}）状態として抽出されます．
また，商品の説明文もそのままの文字列データとして抽出しています．
欠損値（\graybox{NULL}）の穴埋めやフラグ化，説明文の文字数（長さ）への変換といった細かなデータ加工は，後続のPython側で行うため，ここではまず必要なデータを過不足なく結合して引っ張ってくることに専念しましょう．
\begin{figure}[p]
    \begin{minted}[gobble=1,
        bgcolor=background,
        fontsize=\small,
        linenos=false]{sql}

    -- 1つ目の副問い合わせ：出品者ごとの平均評価を算出
    WITH 出品者評価 AS (
        SELECT
            商品.出品者_ユーザID,
            AVG(レビュー.評価スコア) AS 平均評価スコア
        FROM
            商品
        JOIN 取引 ON 商品.商品ID = 取引.商品ID
        JOIN レビュー ON 取引.取引ID = レビュー.取引ID
        GROUP BY 商品.出品者_ユーザID
    ),
    -- 2つ目の副問い合わせ：商品ごとのいいね数を算出
    商品いいね数 AS (
        SELECT
            商品ID,
            COUNT(ユーザID) AS いいね数
        FROM
            いいね
        GROUP BY 商品ID
    )
    -- 予測モデルに必要な特徴量をすべて結合して抽出
    SELECT
        商品.商品ID,
        商品.価格,
        商品.状態,
        商品.説明文,
        カテゴリマスタ.大カテゴリ,
        カテゴリマスタ.小カテゴリ,
        配送方法マスタ.配送方法,
        出品者評価.平均評価スコア AS 出品者平均評価スコア,
        商品いいね数.いいね数 AS 出品者いいね数,
        取引.取引ID
    FROM
        商品
    JOIN カテゴリマスタ ON 商品.小カテゴリ = カテゴリマスタ.小カテゴリ
    JOIN 配送方法マスタ ON 商品.配送方法 = 配送方法マスタ.配送方法
    LEFT OUTER JOIN 出品者評価 ON 商品.出品者_ユーザID = 出品者評価.出品者_ユーザID
    LEFT OUTER JOIN 商品いいね数 ON 商品.商品ID = 商品いいね数.商品ID
    LEFT OUTER JOIN 取引 ON 商品.商品ID = 取引.商品ID;

    \end{minted}
    \captionsetup{name=コード}
    \caption{機械学習向けの特徴量データセットを抽出するSQL文の例}
    \label{code:machine-learning-dataset-sql}
\end{figure}


\section{Pythonによる機械学習モデルの構築}
山畑さんの後輩からは，フリマアプリの将来展望として「商品の売れやすさを予測」が挙げられていました．
本章の締めくくりとして，前節で用いた関係データベース（\graybox{fleamarket.db}）に格納されたデータを用いて，商品の売れやすさを予測する機械学習モデルを構築してみましょう．

\strong{機械学習（Machine Learning）}とは人工知能技術の一つであり，コンピュータと数学を用いて大量のデータに潜むパターンやルールを自動的に発見し，それを分類や予測に役立てる技術です．
機械学習には「教師あり学習」「教師なし学習」「強化学習」などいくつかの問題設定があり，それぞれに合わせて様々な手法が存在します．
「正解/不正解」のようなラベルの付いた大量のデータを用いて，ラベルが付いていないデータのラベルの予想を行うモデルを構築する問題は，\strong{教師あり学習（Supervised Learning）}と呼ばれます．
山畑さんの後輩の展望である，売れた商品と売れ残った商品のデータを用いて商品の売れやすさを予測するという課題は教師あり学習と位置づけることができます．

さて，先にも書いたように，機械学習はデータから法則性を抽出するために，コンピュータと数学の力を使います．
数学と聞くと身構えてしまう方も多いかと思いますが，幸いなことにPythonにはややこしい部分に触れずに，様々な機械学習テクニックを簡単に使うことができるライブラリが公開されています．
後ほど用いるScikit-Learn\footnote{Scikit-Learn: \url{https://scikit-learn.org/stable/}}はその一つです．
ただし，そのような便利ライブラリがあったとしても，すぐに機械学習を始めることはできません．
機械学習の本質は「数学的な計算」であるため，図\ref{fig:to-numerical-data}のように，機械学習の適用対象となるあらゆるデータは数値のあつまり\footnote{正確に言うと，ベクトルや行列，テンソルが入力となる．}に事前に変換しておく必要があります．

分析に適した形にデータを加工，整形，除去する作業も含まれ，分析パフォーマンスを上げるためには前処理が極めて重要となります．

\subsection{欠損値の処理}
データの前処理で典型的に行われる処理の一つが，欠損値の処理です．

多くの機械学習モデルは入力としてベクトルや行列データを想定しており，そこに欠損値があることは想定していません．
ところが，実際のデータには様々な理由でデータの一部が空になっている箇所がある事がよくあります．
先ほど読み込んだデータフレーム\graybox{df}においても，「出品者平均評価スコア」「出品者いいね数」「取引ID」の列が欠損値（NaN）になっているのある行が多数存在します．
なぜなら，まだ評価がない出品者のスコアや「いいね」が1件もついていない商品のいいね数，まだ売れていない商品の取引IDなどは，外部結合時に値が存在しない（NULL）状態として抽出されるためです．
このままでは機械学習を適切に行うことができないため，欠損値を適切に処理する必要があります．

欠損値の処理方法は様々なものがありますが，ここでは以下の方針で対応してみましょう．
\begin{itemize}
\item 出品者平均評価スコア：評価がまだ無いということは「良いか悪いか分からない」ので，中間値である\graybox{3.0}で穴埋めをする
\item 出品者いいね数：いいねが1件もついていないということは「いいね数が0」であると解釈できるので，\graybox{0}で穴埋めをする
\item 取引ID：まだ売れていない商品であることを示すので，\graybox{0}という印をつけておく．逆に取引IDがあるものは\graybox{1}の印をつけておく
\end{itemize}

コード\ref{code:missing-value-handling}は，データフレーム\graybox{df}に対して上記方針で欠損値の処理を行うPythonコードの例です．

\subsection{カテゴリごとの価格の標準化}
続いて，価格の列に対して前処理を行いましょう．
価格は数値データではありますが，商品のカテゴリによって価格帯が大きく異なる可能性があります．
例えば，家電の方が書籍よりも全体的に価格帯が上であることが予想されます．

多くの人にとって値段の安さは魅力の一つです．
価格帯が高い家電カテゴリであっても，高い家電より安い家電は魅力的です，
ところが，価格をそのまま機械学習モデルに学習させてしまうと，「10万円のPC」と「1000円の参考書」をそのまま比較して，モデルが単に「高いものは売れない」というような極端な判断を下してしまう可能性があります．
山畑さんの後輩（そして私たち）が知りたいのは，「売れるカテゴリ」ではなく，同じカテゴリの中で「売れやすい商品」やその特徴です．
カテゴリ内での「お買い得感（相対的な安さ）」をモデルに評価させるために，標準化が必要なのです

この問題を解決するには，異なるカテゴリ間で価格の比較ができるよう，カテゴリごとに価格を標準化します．
図\ref{fig:standardization}のように，\strong{標準化（standardization）}とは，データの分布が平均0，分散1の正規分布に従うように各データを変換する操作です．
数学的には，元データの平均が$\mu$，標準偏差が$\sigma$であるとき，元データの値$x$を以下の式で変換します．
\begin{align}
z = \frac{x - \mu}{\sigma}
\end{align}
標準化も典型的な前処理の1つであり，機械学習のパフォーマンス向上のためによく用いられます．
\begin{figure}[htb]
    \centering
    \includegraphics[width=0.99\textwidth]{figure/design-and-analysis-example/standardization.png}
    \caption{機械学習のための数値データ変換．}
    \label{fig:standardization}
\end{figure}

コード\ref{code:standardization}は，データフレーム\graybox{df}の「価格」列に対して標準化を行うPythonコードの例です．
小カテゴリごとに価格の標準化を行った結果を，新しい列「標準化価格」として追加しています．

\begin{notebox}{なぜSQLではなくPandasで前処理を行うのか}
基本的な集約計算や欠損値の置換などはSQLでも処理はできるのに，なぜ前処理にpandasを用いるのでしょうか．
それは，pandasはPythonの機械学習ライブラリと相性が良く． SQLでは記述が複雑になったり不可能な処理（例：テキスト処理や日付処理，データの縦持ち・横持ち変換，データのスケーリング）を圧倒的に効率よく行えるからです．

一方，pandasにも弱点があります．
それは，メモリに載らないような大規模なデータに対する絞り込みや集計，テーブル間の結合などの処理です．
Pandasは全てのデータを計算機のメモリに載せて処理するため，あまりに巨大なデータを扱うと処理が遅くなったり，最悪の場合処理が止まってしまう可能性があります．
こういった処理は，データベースシステムが得意とする分野で，SQLを用いてデータベースに処理を任せる方が効率的です．

pandasにもSQLにもそれぞれメリットとデメリットがあるため，両者をうまく使い分ける必要があります．
データ分析の現場では，以下のような順で処理をするのがベストプラクティスではないでしょうか．
\begin{enumerate}
    \item SQL使用フェーズ：データベース内のテーブルを結合し，必要に応じて絞り込み，集計演算を行い，Pythonを動かす計算機のメモリに載る程度の量の分析用データセットを抽出する．
    \item pandas/Python使用フェーズ：SQLによって抽出されたレコードをデータフレームに読み込み，手間のかかる前処理を行い，機械学習ライブラリに読み込ませるためのデータを整形する．
\end{enumerate}
\end{notebox}


\subsection{機械学習モデルの構築と評価}
前処理も終わり，学習用のデータセットが完成したので，いよいよ機械学習モデルの構築を行いましょう．
今回は，商品の売れやすさを予測するためのモデルを構築することが目標でした．
この課題設定は，商品が「売れる」か「売れないか」の2値を予測する分類問題として位置づけられます．
分類問題を解くための機械学習モデルには，ロジスティック回帰やサポートベクターマシン，ニューラルネットワークなど様々なものがあります．
ここでは手軽に試せつつ，そこそこ性能の良いランダムフォレスト（Random Forest）を用いてみましょう．

基本的な機械学習アルゴリズムは，PythonのScikit-Learnライブラリを用いて簡単に利用することができます．
Scikit-Learnを用いた教師あり学習問題用の機械学習モデルの構築は，以下のような流れで行います．
\begin{enumerate}
    \item データセットを説明変数（特徴量）と目的変数（ラベル）に分ける
    \item 用いる機械学習モデルを定義する
    \item データセットを用いて機械学習モデルを訓練する
    \item 学習したモデルの性能を評価する
    \item （満足のいく性能が得られたら）構築したモデルを使って，新しいデータの予測を行う
\end{enumerate}

それでは先ほど用意したデータフレーム\graybox{df\_processed}を用いて，この流れに沿って「商品が売れるかを予測するモデル」を構築してみましょう．
まずはステップ1と2です．
このステップについては，pandasとScikit-Learnを使うとわずか数行で完了します．

続いて，モデルの学習と評価を行います．
Scikit-Learnは非常に手軽にモデルの学習を行えるようになっており，\graybox{model.fit(X, y)}というコードを実行するだけで，データ$X$と$y$の対応関係を予測するモデルが構築できます．
しかし，いきなりその手続きを行ってはいけません．
機械学習では，モデルの学習と評価を適切に行うための「お作法」があります．

モデルが構築できたら，どの程度正確に予測ができるかを評価する必要があります．
ここで問題が起きます．
大抵のプロジェクトの場合，プロジェクト開始時にモデル構築用のデータセットは用意されていますが，予測性能を評価するためのデータセットは用意されていません．
機械学習に馴染みのない方であれば，モデル構築に用いたデータセットをそのまま評価にも使えばよいのではと思うかもしれませんが，これは大きな間違いです．
モデルの学習に用いたデータをそのまま評価にも用いてしまうと，モデルが学習用データに過剰に適合してしまっていた場合に，学習データには高い予測性能は発揮できるのに，学習データに含まれない未知のデータはまったく性能が発揮できない，という事態を招いてしまいます．
テストで点を取るために過去問をたくさん勉強した結果，過去問は高得点が解けるようになっていい気になっていたが，本番テストでは点が取れなくて悲しい思いをする -- というのと同じようなことが起きてしまうのです．

こういった事態を避けるために，一般に機械学習を行う際には，図\ref{fig:ml-ops}のように，手持ちのデータを学習用データと評価用データに分割を行い，学習用データを用いてモデルの構築を起こった後，使っていない評価用データを用いてモデルの性能を評価する，という手順を踏むことがお作法となっています．

    \includegraphics[width=0.99\textwidth]{figure/design-and-analysis-example/ml-ops.pdf}

\end{figure}

この学習と評価のプロセスは，データの量によっていくつかのバリエーションがあります．
ここでは，シンプルさを優先して，手持ちのデータセットを学習用データと評価用データに一定の割合に分割する方法（ホールドアウト法）を採ります．
コード\ref{code:model-train-and-evaluation}は，データセットを7対3の割合で学習用データと評価用データに分割し，学習用データを用いてモデルの学習を行い，評価用データを用いてモデルの性能を評価するPythonコードの例です．
「評価データに対する予測性能」の出力結果には，売れると予測された商品のうち実際に売れた商品の割合（適合率; precision），実際に売れた商品のうち売れると予測された商品の割合（再現率; recall），および適合率と再現率の調和平均であるF値などが表示されます．

    from sklearn.model_selection import train_test_split
    from sklearn.metrics import classification_report

    # データセットを学習用データ（70%）と評価用データ（30%）に分割する
    # stratify=yは，目的変数（売れた/売れ残ったのラベル）の割合を

    print(classification_report(y_test, y_pred))

    # model.feature_importances_は，モデルにおける各特徴量の重要度を表す数値の配列を返す．
    importances = pd.Series(model.feature_importances_, index=X.columns)

    # 重要度順に並び替えて，先頭の10件を表示
    print(importances.sort_values(ascending=False).head(10))
